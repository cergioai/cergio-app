SRP STATUS — REAL ACTIVITY, NOT setInterval

User directive (2026-05-28): "make it related to REAL actions (as
opposed to hard wired...)". The status ticker on /results now reads
LIVE counts from the notifications + bids tables for the open
request, advancing only when actual rows land.

New hook: src/hooks/useRequestActivity.js
  - polls notifications WHERE data->>request_id = <id> AND kind='new_request'
  - polls bids WHERE request_id = <id>
  - 2.5s cadence (cheap, well under any rate limit)
  - exports activityToStatus() that turns counts → status line + 0..1 intensity
  - schema-cache miss on bids treated as zero (graceful while we wire bids)

ResultsScreen wiring:
  - useRequestActivity(chatState.request_id) at the top
  - liveStatus.line replaces scripted line as soon as a real row lands
  - Scripted lines (setInterval) kept as graceful fallback for the
    ~3s pre-write window
  - Sub-line surfaces "X replies in — Y notified" when bids arrive

LEAF INTENSITY (canonical formula in activityToStatus)
  intensity = min(1, (notified + replied * 3) / 10)
  bids count 3× notifications — a reply is real engagement, a
  notification is just a ping. Ready to consume from LeafLogo when
  the next pass wires the prop through.
