fix(referral): credit 7% slice per booking (not lump $250 cap)

Tarik 2026-06-05 on the live earnings page: "data is hardcoded — can't
have $250 max reached for several rows when only 1 user invited."

The symptom traced to `creditInviterOnFirstBooking` in
src/lib/referral.js. The old implementation wrote a SINGLE earnings
row of `REWARDS.perFriend * 100` cents ($250) per invitee's first
booking — ignoring the actual booking total + the 7% share. Result:
every new invitee's first booking inflated the inviter's earnings by
$250, regardless of whether the booking was $5 or $5000.

That collided with the v6 cap-walk logic (EarningsScreen rowCapState)
which buckets rows by friend → showing "$250 cap maxed" on EACH
friend's row even though the actual booking economics never produced
that much.

══════════════════ Fix ════════════════════════════════════════════════

  src/lib/referral.js — creditInviterOnFirstBooking now:

  1. Looks up the actual booking total_cents from the bookings table.
  2. Sums prior direct earnings for this (inviter, invitee) pair so
     the cap-room calc is correct on re-fires.
  3. Computes the credit as:
        credit = min(7% × bookingTotal, $250 - prior)
  4. Resolves the invitee's display_name and stamps it on the
     earnings row's meta.friend — so EarningsScreen's cap-walk
     (rowCapState) groups rows by friend name correctly.
  5. Skips the earnings insert when credit = 0 (booking $0 or cap
     already reached prior), but still stamps first_booking_at so we
     don't re-fire endlessly.

  Net effect: a single $95 booking from Daniela now produces a
  $6.65 earnings row for Camilo (the inviter) — not a $250 lump.
  Multiple bookings accumulate up to the $250 cap as designed.

══════════════════ Seed fixture cleanup ═══════════════════════════════

  Seed Dense Network.command — DELETE phase extended to:

  • Wipe earnings rows whose source_id IS IN the dense invites
    (clears the stale $250 lumps the old code wrote).
  • Reset first_booking_at + reward_cents on dense invites so the
    next booking event re-fires the credit through the new logic.

  Tarik to re-run Seed Dense Network.command after this pushes; then
  the existing demo bookings will re-trigger creditInviterOnFirstBooking
  with the correct 7%-slice economics.

══════════════════ Notes ══════════════════════════════════════════════

  • Does NOT touch creditChainOnFirstBooking ($12.50 chain bonus).
    That contract is a fixed bonus, not a percentage — unchanged.
  • Re-running on existing booked invites is safe; the prior-cents
    lookup prevents over-credit. The reset of first_booking_at in the
    seed is what makes the demo data refresh.
