Fix: friends-on-search invisible to Tarik (RLS) + Earnings empty (no seed rows)

Tarik tested live: Penny's card showed "No mutual friends yet" even
though recommendations were seeded, and Earnings was empty. Two
distinct causes — both fixed by re-running Seed Money Flow Test Data.

ROOT CAUSE A — recommendations RLS too restrictive

The recommendations table's existing SELECT policy almost certainly
gates on `auth.uid() = recommender_id` (standard "self-only" pattern).
That means Tarik's authed session can only see HIS OWN recommendations,
never the ones Alex / Sam / Connie wrote about Penny's service —
even though the rows are in the DB.

Recommendations are essentially social-proof / marketing data — they're
meant to be visible to potential consumers. Without a permissive read
policy, the FriendAvatars stack + ConsumerPDP recommender panel
always render empty for everyone except the recommender themselves.

Fix: the seed now appends a SQL block that creates a permissive
`recommendations read all` policy via the Supabase Management API:

  alter table recommendations enable row level security;
  drop policy if exists "recommendations read all" on recommendations;
  create policy "recommendations read all" on recommendations
    for select using (true);

Safe because recommendations are public-by-design; write policies
(insert/update) remain restrictive (auth.uid = recommender_id).

ROOT CAUSE B — no seeded earnings rows

The Earnings tab was empty because the seed only created personas +
services + recommendations, never earnings rows. No bookings have
flowed money, so the ledger had nothing to render.

Fix: seed now inserts 7 representative earnings rows so the Direct /
Chain +5% tier pills render against real data:

  Tarik  $250   direct    "friend":"Alex"
  Tarik  $250   direct    "friend":"Connie"
  Tarik  $63    direct    "friend":"Sam"  (partial cap, mid-accumulation)
  Tarik  $12.50 fof       "friend":"Sam->FriendA"
  Tarik  $12.50 fof       "friend":"Sam->FriendB"
  Connie $150   spotlight (instagram)
  Penny  $279   booking   (Penny's Plumber, $300 booking minus 7% fee)

meta.tier carries 'direct' or 'fof' explicitly so the earningTier
classifier uses the field directly (not the amount fallback). Re-running
the seed is safe — `on conflict do nothing` covers the unique-by-id
constraint and source_ids are generated per row via gen_random_uuid.
