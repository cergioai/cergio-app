Fix: /inbound Accept works from a direct link (resolve own service)

Bug: accepting a connector request said "you need a listed service" even
when the provider HAS a matching listing — because RequestFromConnectorScreen
relied entirely on the ?myServiceId= param that only the Inbox card adds. A
direct/shared /inbound link had no param, so Accept/Counter always failed.

Fix: when the URL has no myServiceId, resolve the signed-in provider's own
matching LISTED service (taxonomy_provider_type → category → title match
against the request type, else the only/most-recent listing). All respond
handlers now use effectiveServiceId = myServiceId || resolved. listMyServices
now also returns taxonomy_provider_type for the match. qa #48 green.
