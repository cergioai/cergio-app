4. GUARDRAILS — pre-push qa gate + houskeeper-bug regression locks

Tarik: "regarding the housekeeper issue earlier, need to ensure
guardrails are in place so these don't fail... we just can't have a
moving target... implement superior processes to deliver first class
platform."

Two new static invariants + a pre-push qa gate that aborts the push
if qa fails. The houskeeper bug (where "need houskeeper under 200"
fell into the "What service do you need?" empty state even though
local taxonomy correctly resolved House Cleaner) is now locked at the
gate level: any revert that re-introduces the bug fails before merge.

  scripts/qa.mjs #35 chat-reply-gate (new)
    • Statically asserts useChat backfills merged.what from a locally-
      resolved provider_type (whatFromTaxonomy)
    • Asserts the reply branches on whatKnown = (what || provider_type),
      NOT on !merged.what alone

  scripts/qa.mjs #36 actor-noun-integration (new)
    • Simulates the chat-reply gate against 10 realistic actor-noun
      queries with typos (houskeeper, plummer, electrcian, sitter, etc.)
    • For each, asserts whatKnown === true so the user would never see
      the "What service do you need?" empty-state regression

  Unlock and Push.command
    • Pre-push gate: runs `node scripts/qa.mjs` BEFORE staging the
      commit. If qa fails, aborts with a clear "fix locally, re-run"
      message. Emergency override: SKIP_QA=1
    • This is the "superior process" — no moving target on critical
      flows, no broken main, no silent regressions
