═══════════════ Step 1: Schema (ships as .command) ═══════════════

`Apply Request Responses Migration.command` (sibling of Unlock and
Push.command) creates the table + 3 indexes + 5 RLS policies.
Idempotent: CREATE TABLE IF NOT EXISTS, DROP POLICY IF EXISTS before
CREATE.

  Table: request_responses (12 columns)
    id                       uuid PK
    request_id               uuid REFERENCES requests(id) ON DELETE CASCADE
    responder_id             uuid REFERENCES profiles(id)
    service_id               uuid REFERENCES services(id)
    status                   text  CHECK IN (offered,countered,declined,
                                              withdrawn,accepted,expired)
    offered_price_cents      int   CHECK >= 0
    message                  text
    responded_at             timestamptz DEFAULT NOW()
    last_counter_by          text  CHECK IN (provider, consumer)
    time_to_offer_seconds    int   (telemetry for time-decay ranking)
    wave_n                   int   (broadcast wave attribution)
    UNIQUE (request_id, responder_id, service_id)

  Indexes: request_id · responder_id · status

  RLS policies (5):
    request_owner_can_select_responses
    responder_can_select_own
    responder_can_insert_own
    responder_can_update_own
    owner_can_accept_or_expire

  Anon role: cannot read. Verified by the migration's sanity step.

═══════════════ Step 1: Client API (lib/api.js) ═══════════════

Four core helpers + one inbound-list helper for the provider inbox:

  respondToRequest(requestId, {status, serviceId, offeredPriceCents,
                               message, waveN})
    — Upserts on (request_id, responder_id, service_id) so repeated
      clicks don't duplicate. Stamps last_counter_by='provider' on
      counters and computes time_to_offer_seconds from the request's
      created_at for time-decay telemetry.

  listResponsesForRequest(requestId, {limit})
    — Consumer-side read: joins responder profile + service. Filters
      to status IN (offered, countered, accepted).

  counterRequestResponse(responseId, {offeredPriceCents, message})
    — Consumer counters back. last_counter_by='consumer'.

  acceptRequestResponse(responseId)
    — Flips chosen → 'accepted', bulk-withdraws siblings.

  listInboundRequests({limit})    ← NEW in Step 2
    — Provider-side read: pulls open consumer requests whose
      service_type matches one of the provider's listed services AND
      which the provider hasn't already responded to. Returns rows
      shaped {id, service_type, description, location_text,
      created_at, requester:{id, display_name}, my_service_id}.
      "Open" = pending status AND created in the last 60 minutes
      (broadcast window per MARKETPLACE_SPEC § 6 S3).

═══════════════ Step 2: Provider-side inbox (JobsInboxScreen) ═══════════════

  • New "New requests near you" section pinned ABOVE the existing
    booking-request list in the Requests tab. Pulls
    listInboundRequests every 30s.

    description, location, time elapsed.

  • Empty / no matching services → no section rendered (silent).
