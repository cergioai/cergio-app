Polygon DB persistence + consumer-side filter + Vite crash fix

Two things bundled — the polygon end-to-end slice and a hot-reload
crash that was masking it.

(1) VITE CRASH FIX — DYNAMIC MAPS LOADER IMPORT
After the package.json bumped @googlemaps/js-api-loader, any dev
server that hadn't run `npm install` choked on the static import in
ServiceAreaMapPicker.jsx — Vite's import-analysis crashes the WHOLE
app on the first load when the package is missing, including
/logo-lab. Tarik reported the lab was blank with this error.

Fix: convert the import to a lazy dynamic import inside
ensureGoogleMaps(). If the package is missing, only the picker
itself fails (with a friendly "run npm install" message); the rest
of the app loads normally.

(2) POLYGON DB PERSISTENCE
ALTER TABLE services
  ADD COLUMN IF NOT EXISTS service_area_geojson jsonb;
Migration in "Apply Service Area Migration.command" (mgmt API via
the same pattern as Apply Follower Count Migration). Idempotent.

  src/lib/api.js
    createService now persists draft.serviceAreaGeoJson on the
    services row. Same defensive retry pattern as taxonomy_* — if
    PostgREST schema cache rejects the column (legacy deployment
    that hasn't run the migration yet), retries without it so the
    listing still publishes.

  src/App.jsx
    listingDraft default + reset both add serviceAreaGeoJson: null.

  src/screens/ServiceListAboutScreen.jsx
    + serviceAreaGeoJson state, areaPickerOpen state
    + "Draw your service area (optional)" button below the Service
      location field. Disabled until coords/location are set
      (the map needs an anchor).
    + Mounts ServiceAreaMapPicker as a bottom-sheet when open.
    + onNext forwards the polygon into updateListingDraft so
      createService picks it up downstream.

(3) CONSUMER-SIDE FILTER
  src/lib/geo.js  (new) — pointInRing / pointInPolygon helpers.
    Ray-casting (Franklin algorithm) on GeoJSON Polygon coordinates.
    Sufficient for city-scale polygons; not great for nationwide or
    antimeridian-crossing shapes (note in the file). No PostGIS,
    no RLS surface, no extension to enable.

  src/lib/api.js listServices proximity branch
    + selects service_area_geojson in the hydrate query
    + After budget/freeOnly filter, drops any service whose polygon
      is set AND whose polygon does NOT contain the consumer's
      (lat, lng). Services without a polygon are unaffected — the
      existing services_near radius logic continues to govern them.

When to swap for PostGIS: once we cross a few thousand providers
returned per query, or once we need server-side filtering (so the
DB scoring/ordering knows about the polygon). For now the client-side
JS ray-cast is fine and adds zero infra.
