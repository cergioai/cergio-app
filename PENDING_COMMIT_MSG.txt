═══════════════ 1. Personal Chef now in dropdown ═══════════════

`src/data/providerTypes.js`:
  + 'Personal Chef' (was missing — only 'Private Chef' existed).

═══════════════ 2. Inline taxonomy hint ═══════════════

`ServiceListAboutScreen` now surfaces the result of the already-wired
`useTaxonomyResolve` hook directly under the Service Type input. Three
states, three UI weights — all NON-blocking (the provider can still
submit; we just steer them):

  • RESOLVING ("Checking taxonomy…") muted spinner copy.
  • MAPPED (chat-parse confidence ≥ 0.60 AND offering_id present)
    Mint pill: "Mapped to {provider_type} · related: {top 3
    candidate offering names}". So when a provider types
    "Personal Chef" we show "Mapped to Personal Chef · related:
    Vegetarian Meal Prep, Sushi Chef, Weekly Meal Prep" inline,
    exactly the "augment the taxonomy with related terms" pattern
    Tarik asked for.
  • LOW-CONF (resolver couldn't classify confidently AND the text
    isn't in the canonical PROVIDER_TYPES list)
    Amber warning: "We don't recognize {X} as a known service
    type. Try something more specific (e.g. Personal Chef,
    Plumber). You can still submit — Cergio will review it."
    Tomato / big burger / random nonsense hits this path.

═══════════════ 3. Novel-type telemetry (gradual backfill) ═══════════════

On submit, if the typed value isn't an exact match against either
the STARTER_TYPES short list or the canonical 373-entry PROVIDER_TYPES
catalog, we log a structured payload so Tarik can grep:

  console.warn('[cergio/novel-provider-type]', {
    novel_type:          '<typed value>',
    resolved_to:         '<chat-parse picked provider_type>',
    confidence:          0.71,
    accepted:            true,
    candidate_offerings: ['Vegetarian Meal Prep', 'Sushi Chef', …],
    source:              'list-service-about',
    ts:                  '2026-06-02T20:14:33Z',
  });

Plus best-effort persistent log via `logEvent('novel_provider_type',
payload)` if/when `src/lib/api.js` ships that helper — currently a
silent no-op (try/catch) so the new path is forward-compatible
without blocking on the helper landing first. The console channel is
the deliverable for this turn; a follow-up commit can ship a
`provider_type_suggestions` table + an admin "review novel types"
view to surface what's piling up.

═══════════════ 4. What this DOESN'T do (deferred) ═══════════════

  • No hard reject. Even "tomato" submits; the listing just goes
    in with taxonomy_provider_type=null, so consumer search won't
    surface it via type — but the row exists, and the warning told
    the provider why. Hard reject would block legitimate novel
    types (this exact bug Tarik just hit with Personal Chef).
  • No backend taxonomy mutation. The novel-type log is read-only
    until we ship the suggestions table + a backfill PR flow.
  • Related-offering suggestions come from the chat-parse resolver's
    `candidates` array. Wiring an "Add this related offering"
    one-tap path is the natural next step; for now it's display only.
