Fix: recommendations.sent_at (not created_at) — the friends-on-search root cause

THE actual root cause of "No mutual friends yet" on every search card,
even when the seed claims to have written 3 recommendations on Penny.

Caught by Diagnose Money Flow.command. The Diagnose output revealed:

  ─── 1a. recommendations columns ───
  [
    {"column_name":"id"},               {"column_name":"recommender_id"},
    {"column_name":"recipient_id"},     {"column_name":"recipient_phone"},
    {"column_name":"service_id"},       {"column_name":"message"},
    {"column_name":"sent_at"},          {"column_name":"inviter_id"}
  ]

  ─── 3c. recommendations on Penny services ───
  {"message":"Failed to run sql query: ERROR: 42703: column r.created_at
   does not exist...HINT: Perhaps you meant 's.created_at'."}

The recommendations table uses `sent_at`, NOT `created_at`. My
fetchRecommendersByServiceId helper (and the PDP cold-fallback) both
queried `created_at` → PostgREST returned an error → Supabase client
wrapped it as { data: null, error } → my helper returned {} → every
ProviderCard's `friends` array was empty → "No mutual friends yet".

Diagnose section 4 confirmed: both SERVICE_KEY and ANON_KEY CAN read
the recommendations (RLS is fine — 3 PERMISSIVE SELECT policies OR'd
together, with `qual: true` winning). The only blocker was the wrong
column name in the client-side query.

THE FIX

  src/lib/api.js (fetchRecommendersByServiceId)
    • SELECT sent_at (not created_at)
    • ORDER BY sent_at (not created_at)
    • Alias `sent_at` → `created_at` in the returned object so the
      downstream consumers (ResultsScreen, PDP) keep their existing
      field name without churn

  src/screens/ServiceDetailScreen.jsx (cold-deep-link fallback)
    • Same fix: SELECT + ORDER on sent_at, alias on the way out

  scripts/qa.mjs #37
    • New static lock: api.js + PDP MUST NOT reference
      recommendations.created_at. Regex-checks both files for any
      .select() with created_at OR .order('created_at' near a
      from('recommendations'). Catches a reversion at the gate.

WHY THIS HID FOR SO LONG

  • Supabase JS client swallows PostgREST errors silently when the
    caller doesn't check `error` — my helper returned {} on error
    which looked identical to "no rows" in the UI
  • The Diagnose script DID call this out earlier (section 3c errored
    the same way) but I missed it — I was focused on the RLS theory
  • Building a [CERGIO/recommenders] DevTools log earlier helped, but
    the user's browser cache hid the change. Hard refresh was needed
    to see the error.

The diagnostic infrastructure paid off: without Diagnose Money Flow's
schema dump (section 1a), I'd still be guessing.
