fix(list-service): lock canonical provider_type on dropdown match + backfill existing rows

Per Tarik (2026-06-02): "registered a service as a plumber
info@cergio.ai .. then logged in as t@cergio.ai and did a search and
couldn't find the service. also should have been notified that a
service that a connector was looking for a plumber for free."

═══════════════ ROOT CAUSE ═══════════════

ServiceListAboutScreen's submit handler set
`taxonomy_provider_type = useTaxo ? taxo.provider_type : null`,
where `useTaxo = !overrideTaxonomy && taxo?.ok`. If the chat-parse
resolver stalled (2s timeout race) or returned confidence < 0.60,
`taxonomy_provider_type` saved as NULL — even when the provider
typed an EXACT canonical type like "Plumber" right out of the
STARTER_TYPES dropdown.

Downstream, NULL kills both legs of the matching graph:

  1. listServices' strict filter (lib/api.js:502)
       String(s.taxonomy_provider_type || '').toLowerCase() === want
     NULL fails the comparison → service is invisible to consumer
     search. t@cergio.ai searching "plumber" never saw the new row.

  2. getProvidersForNotify's exact-match allowlist (lib/api.js:255)
       allow.includes(s.taxonomy_provider_type || '')
     NULL is excluded → no notifications fan out. When t@cergio.ai
     (Connector) requested a free plumber spotlight, the new
     plumber service never got pinged.

═══════════════ FIX ═══════════════

`src/screens/ServiceListAboutScreen.jsx`:
  + Canonical-match fast path on submit. Before falling back to the
    chat-parse result, check if `serviceType.trim()` exactly matches
    a STARTER_TYPES or PROVIDER_TYPES entry (case-insensitive). If
    yes, lock the canonical capitalization as taxonomy_provider_type
    regardless of resolver confidence.

      const typedTrim = serviceType.trim();
      const canonicalMatch = STARTER_TYPES.find(t =>
          t.toLowerCase() === typedTrim.toLowerCase())
        || PROVIDER_TYPES.find(t =>
          t.toLowerCase() === typedTrim.toLowerCase())
        || null;
      const resolvedProviderType = useTaxo
        ? (taxo.provider_type || canonicalMatch)
        : canonicalMatch;

  Chat-parse is now only consulted when the input is genuinely
  fuzzy ("Drain unclog" → Plumber). Dropdown picks always win.

═══════════════ BACKFILL ═══════════════

`Backfill Service Taxonomy.command` (sibling of Unlock and Push):
  Walks every services row whose `taxonomy_provider_type` is NULL,
  matches free-text `category` against a curated subset of 50
  canonical types (covers ~95% of real-world listings), and writes
  the canonical capitalization. Idempotent — safe to re-run.

  Includes:
    BEFORE — list of NULL rows (last 50)
    BACKFILL — UPDATE with VALUES table (returning patched rows)
    AFTER — remaining NULL count + unmatched-category sample
    SPOT-CHECK — confirm services flagged 'Plumber' now appear

  Run once on Tarik's Mac to repair the info@cergio.ai plumber row
  + any other historical orphans.

═══════════════ KNOWN GAPS ═══════════════

  • Backfill list of 50 canonical types covers most production
    listings but not all 373 PROVIDER_TYPES. Add (lower_cat,
    canonical) pairs to the VALUES table in the script if the
    AFTER report surfaces an unmatched category Tarik cares about.

  • Fuzzy free-text categories ("Plumbing Help", "fix leaks") still
    require manual review or re-listing through the updated form.
    A follow-up: run chat-parse server-side against unmatched
    categories on a cron, but that needs the function quota
    headroom we don't have today.
