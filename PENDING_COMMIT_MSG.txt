Maps via <script>, smoother share message, drop @googlemaps npm dep

Bundled fixes — both blocking Tarik's testing session.

(1) MAPS LOADER → <script> TAG (no npm dep)
Vite's dev-mode import-analysis ignores the /* @vite-ignore */ pragma
on dynamic imports. Tarik hit the same "Failed to resolve" overlay
twice after I patched it. Real fix: stop importing the npm package
entirely. Inject a <script src="https://maps.googleapis.com/maps/
api/js?..."> on demand from a module-level promise. Zero npm
dependency — Google's loader resolves + caches itself, Vite has
nothing to analyze. Removed @googlemaps/js-api-loader from
package.json. Reinstall Deps and Start.command is no longer needed
for the map to work (still useful for any future dep adds).

(2) SHARE MESSAGE GRAMMAR
Tarik flagged the share text from the empty state read awkward:
  OLD:
    "Hey — anyone know a good house cleaner — This week, in 5700
     Collins Ave, Miami Beach, FL 33140, USA? Booking on Cergio →
     <url>"
  NEW:
    "Hey — anyone know a good house cleaner? Need one this week in
     5700 Collins Ave. Budget around $50. Booking on Cergio →
     <url>"

Two locations were building this string with slightly different glue
(header share button + empty-state share card). Consolidated into
one module-level helper `buildShareMessage({lead, when, where,
budget, details, inviterUrl})` so both surfaces stay in sync.

Specific changes inside the helper:
  • Address truncated to first comma segment ("5700 Collins Ave"
    not "5700 Collins Ave, Miami Beach, FL 33140, USA") — recipients
    don't need ZIP+country noise
  • Budget ALWAYS included when set. Prior code skipped budget if
    its text appeared anywhere in user's query — too aggressive,
    dropped real budget signals
  • Sentence rhythm: "Hey — anyone know a good {X}?" / "Need one
    {when} in {where}." / "Budget around {budget}." / link
  • "Need one {when}" reads naturally without "for" — works for
    "this week", "tomorrow", "ASAP", "Friday morning", etc.
