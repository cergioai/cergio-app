fix(list-service): never save generic taxonomy_provider_type

═══════════════ The bug ═══════════════

Tarik 2026-06-03:
  "sent a plumber request with t@cergio.ai .. logged in as
   info@cergio.ai .. didn't find any notification."

Notify-chain audit surfaced the actual cause: info@cergio.ai's TWO
Personal Chef listings both carried taxonomy_provider_type=
'Service Provider' — the generic catch-all string. Both
getProvidersForNotify (fan-out allowlist) and listInboundRequests
(provider's open-requests inbox) do an EXACT-match against the
canonical taxonomy. Generic values silently exclude the provider
from every fan-out AND every search result.

Same root cause as the NULL-taxonomy bug shipped in a5909e6
(2026-06-02). a5909e6 fixed NULL → canonical via the dropdown
fast path; it did NOT guard against chat-parse returning a
generic non-NULL string. So a stalled / drifted parser response
could still poison the row.

═══════════════ Source fix (ServiceListAboutScreen) ═══════════════

  • GENERIC_PT set + isGeneric() guard.
  • When useTaxo is true but taxo.provider_type is generic, fall
    through to canonicalMatch (the dropdown selection).
  • Net behavior: resolvedProviderType is either
      - a canonical PROVIDER_TYPES entry from the parser, OR
      - the canonical dropdown selection, OR
      - null (handled downstream — never a generic catch-all).
  • Matches the GENERIC_PROVIDER_TYPES set ResultsScreen
    already uses to skip generic values in display copy.

═══════════════ Data fix (one-shot .command) ═══════════════

`Fix Generic Taxonomy.command` walked every services row whose
taxonomy_provider_type was in the generic set and remapped to
canonical via title / category keyword. Ran clean: 2 rows
remapped (info@cergio.ai's Personal Chef pair), 0 remaining
generic, spot-check confirms they're now visible to Personal
Chef searches + fan-outs.
