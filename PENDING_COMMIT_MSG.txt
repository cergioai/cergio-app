Search: bulletproof typo/variant/Spanish tolerance + softer SRP reco card

BACKGROUND  User typed "deep clenaings" (typo of "deep cleanings") and
the SRP rendered "No deep clenaings yet — ask friends" + a share
message that read "anyone know a good deep clenaing to deep clenaing".
Two bugs surfaced:

  1. resolveProviderTypeLocal didn't tolerate typos at all — exact
     substring match only.
  2. When the parser failed to resolve a canonical type, the share
     message fell back to the user's raw verb-phrase TWICE in the
     same line, producing duplicate gibberish.

User directive: "search match notification can't fail on silliness …
build elegant tolerance and imagine all possible scenarios."

FUZZY MATCHER (src/lib/serviceTaxonomy.js)

  Added a 2-pass resolveProviderTypeLocal:

    Pass 1 (canonical): exact substring match. Same as before.
    Pass 2 (fuzzy):     Levenshtein edit-distance fallback.
                        Tokens ≤3 chars  → exact match required
                        Tokens 4–5 chars → ≤1 edit
                        Tokens 6–7 chars → ≤2 edits
                        Tokens ≥8 chars  → ≤3 edits

  ALL meaningful tokens in a taxonomy key must fuzzy-match SOME user
  token. Longest matching key wins (same precedence as Pass 1).
  Conservative — never overrides an exact match, never matches
  nonsense like "random nonsense" or "asdfghjkl".

SHARE-LEAD GRAMMAR GUARD (ResultsScreen)

  Old bug: "anyone know a good plumber to toilet unclogged" — the
  shareAction was a noun-led phrase that read poorly after "to".

  Fix: only include the action when its FIRST token is in a known
  ACTION_VERBS whitelist (unclog, clean, fix, walk, cook, train,
  teach, mount, etc.). Noun-led queries fall back to just the
  canonical noun: "anyone know a good plumber" — clean read.
