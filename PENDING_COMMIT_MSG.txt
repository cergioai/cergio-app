fix(list-service): blank address for new providers (welcome resets listingDraft)

Per Tarik (2026-06-01): "registered as a new service and saw a default
address (5700 collins). should be blank for new users."

Root cause: `listingDraft` lives in App.jsx-level state for the entire
session. If the user previously typed an address (or any prior in-
progress listing pass through Welcome populated the draft), the
ServiceListAboutScreen pre-filled `location` from the stale value:

  const [location, setLocation] = useState(listingDraft.location || '');

New users see a blank field on a fresh boot, but Tarik's flow had
prior chat-state leakage from earlier testing — the form felt sticky
and confusing.

Fix:
  • ServiceListWelcomeScreen pulls `resetListingDraft` from outlet
    context and calls it before navigating to /list-service/about
    via the "List my service" CTA.
  • Welcome IS the canonical "start fresh" entry — resumed flows
    would route directly to /list-service/about without passing
    through Welcome (which already works since the draft persists).

No schema changes; no API changes.

Verified:
  qa.mjs — 41/41 PASS
  acorn-jsx parse on ServiceListWelcomeScreen.jsx — GREEN.

To test on Tarik's Mac: tap "List a service" anywhere in the app,
hit "List my service" on the welcome screen, and the Service
Location field should be empty. Type a new address and proceed —
listingDraft populates from the typed value, not from saved state.
